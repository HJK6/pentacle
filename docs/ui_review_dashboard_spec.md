# Pentacle UI Review Dashboard Spec

## Objective

Build a repeatable UI review system that lets any Triforce machine publish visual QA artifacts and lets any Pentacle instance view them automatically.

The system must support future screens, assets, and repos without requiring a Pentacle desktop app update. Pentacle should be a generic artifact browser; repos and publishers own the content.

## Scope

- Static UI review artifact bundles generated by any repo.
- Cross-machine discovery through the existing Dashboard Hub.
- Generic Pentacle dashboard for browsing, filtering, and previewing artifacts.
- Local-file scanning as a development fallback only.
- Pentacle mobile chat UI options as the first concrete artifact producer.

## Non-Goals

- Pentacle should not contain repo-specific rendering logic.
- Pentacle should not need a source or app update when a repo adds a new screen, option board, screenshot, or asset.
- This does not replace unit tests, integration tests, device tests, or app release validation.
- This does not require public internet hosting; Tailscale/private hub serving is enough.

## Architecture

```text
repo generator
  -> .ui-review/<artifact>/manifest.json + index.html + assets
machine publisher
  -> uploads artifact bundle to Dashboard Hub
Dashboard Hub
  -> stores bundle
  -> serves artifact files over HTTP
  -> publishes ui.review.index envelope over existing websocket
Pentacle
  -> reads ui.review.index
  -> renders generic artifact browser
  -> previews artifact entry URL in iframe
```

## Artifact Contract

Each artifact is a directory under `.ui-review/<artifact-id>/` or another configured artifact root.

Required files:

- `manifest.json`
- an entry HTML file named by `manifest.entry`, usually `index.html`

Required manifest fields:

```json
{
  "schema": "pentacle.uiReview.v1",
  "id": "pentacle-mobile/mobile-chat-color-options",
  "repo": "pentacle-mobile",
  "title": "Mobile Chat Color Options",
  "entry": "index.html",
  "createdAt": "2026-04-27T00:00:00Z"
}
```

Optional manifest fields:

```json
{
  "summary": "Machine color treatments for chat list and open chat header",
  "machine": "merlin",
  "tags": ["mobile", "chat", "color"],
  "screen": "mobile-chat",
  "source": {
    "repoPath": "/Users/vgujju/repos/pentacle-mobile",
    "gitCommit": "abcdef1",
    "dirty": true,
    "generator": "npm run mockups:chat-ui"
  }
}
```

Pentacle must ignore unknown fields. Future schema versions should be additive where possible.

### Manifest Validation Rules

- `schema` must equal `pentacle.uiReview.v1` for v1 publishers.
- `repo`, `id`, `title`, and `entry` must be non-empty strings.
- `repo`, `id`, and `machine` must be treated as identifiers, not filesystem paths.
- `entry` must be a relative path within the artifact bundle.
- `entry` must not be absolute, contain `..`, or resolve outside the artifact root.
- Symlinks must not be followed when packaging or serving bundles.
- Unknown manifest fields must be preserved in metadata but ignored by Pentacle's generic browser.
- Invalid artifacts must be skipped without blocking other valid artifacts from the same scan.

Canonical artifact key:

```text
artifactKey = slug(repo) + "/" + slug(id) + "/" + slug(machine)
```

`slug` lowercases, trims whitespace, replaces non `[a-z0-9._-]` characters with `-`, collapses repeated `-`, and rejects empty output.

The machine segment is part of the key so the same repo/artifact can be published independently by Bartimaeus, Merlin, and Amaterasu without collision. The latest artifact for a given key replaces older content at the same served URL.

## Publisher Contract

Each machine runs a publisher that scans configured roots:

- `~/repos/*/.ui-review/*/manifest.json`
- `~/repos/*/test/artifacts/*.html` as a legacy compatibility source
- `~/agent-workspace/ui-review/*/manifest.json`

For each artifact, the publisher:

- validates the manifest
- resolves `entry`
- computes the canonical artifact key from `repo + id + machine`
- captures machine name, updated time, file list, sizes, and source git state when available
- uploads the bundle to Dashboard Hub
- republishes when content changes

Publishing should be idempotent. If the same artifact ID is republished by the same repo/machine, the hub replaces that artifact version and keeps the URL stable.

Publisher defaults:

- max bundle size: 25 MB
- max file count: 500
- allowed file types: `.html`, `.css`, `.js`, `.json`, `.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.svg`, `.txt`, `.map`
- scripts are allowed inside artifacts because many UI review pages need local interactivity, but they run inside a sandboxed iframe in Pentacle

## Dashboard Hub Contract

Dashboard Hub owns cross-machine distribution.

Required API:

- `POST /ui-review/publish`
  - authenticated with an existing or new write token
  - accepts manifest plus zipped artifact bundle
  - validates path safety and manifest schema
  - stores the bundle under a hub-owned artifact directory

- `GET /ui-review/artifacts/<artifact-key>/<path>`
  - serves uploaded artifact files
  - rejects path traversal
  - serves only files inside the stored bundle
  - emits `Content-Security-Policy` that allows same-artifact scripts/styles/images and blocks top navigation

Hub base URL:

- Dashboard Hub must publish absolute artifact URLs in `entryUrl`.
- URLs must be reachable from Bartimaeus, Merlin, and Amaterasu over Tailscale or an equivalent private route.
- `http://hub` in examples is a placeholder; deployed configs must use the real Dashboard Hub HTTP base URL.

Required dashboard envelope:

```json
{
  "dashboard_id": "ui.review.index",
  "data": {
    "schema": "pentacle.uiReviewIndex.v1",
    "generatedAt": "2026-04-27T00:00:00Z",
    "artifacts": [
      {
        "artifactKey": "pentacle-mobile/mobile-chat-color-options/merlin",
        "id": "pentacle-mobile/mobile-chat-color-options",
        "repo": "pentacle-mobile",
        "machine": "merlin",
        "title": "Mobile Chat Color Options",
        "summary": "Machine color treatments for chat list and open chat header",
        "tags": ["mobile", "chat", "color"],
        "updatedAt": "2026-04-27T00:00:00Z",
        "entryUrl": "http://hub/ui-review/artifacts/pentacle-mobile/mobile-chat-color-options/merlin/index.html",
        "source": {
          "gitCommit": "abcdef1",
          "dirty": true
        }
      }
    ]
  },
  "freshness_ttl_sec": 300
}
```

## Pentacle Dashboard Contract

Pentacle's `UI Review` dashboard must:

- read `ui.review.index` from Dashboard Hub through the existing hub client
- render repo, machine, tag, and title metadata generically
- preview `entryUrl` in an iframe
- provide loading, empty, stale, and error states
- show source dirty/git state when present
- in packaged/normal use, prefer stale cached hub data over local fallback
- use local-file scanning only when hub data is unavailable and the app is running in development or `uiReview.localFallback` is explicitly enabled

Pentacle must not require changes for new artifacts, screens, assets, repos, or option-board layouts.

Iframe preview:

- Use `sandbox="allow-scripts allow-same-origin"` for hub-served artifact previews.
- Do not grant `allow-top-navigation`, `allow-popups`, or form permissions in the preview iframe.
- The Open button may open the artifact URL outside the sandbox using Electron/browser behavior.

## UX Requirements

- Default view shows the latest artifact first.
- List rows show title, repo, machine, updated time, and dirty state.
- Filters are data-driven and optional; absence of filters must not block browsing.
- Preview area must remain usable for desktop-sized and phone-sized artifacts.
- Open-in-browser should use the served hub URL when available.

## Acceptance Criteria

- A new artifact generated in `pentacle-mobile` appears in Pentacle without adding dashboard-specific code.
- A new artifact with arbitrary HTML/CSS/assets appears without a Pentacle source change.
- An artifact published on Merlin can be viewed from Pentacle on Bartimaeus or Amaterasu.
- An artifact published on Bartimaeus can be viewed from Pentacle on Merlin or Amaterasu.
- An artifact published on Amaterasu can be viewed from Pentacle on Merlin.
- When Dashboard Hub is offline, Pentacle shows stale cached hub data with a clear stale/offline status.
- Local fallback appears only in development or when explicitly enabled.
- The current Pentacle mobile chat QA artifact remains viewable.

## Extensibility Acceptance

After initial deployment, create a new artifact that Pentacle has never seen before:

- new repo or repo not hardcoded by Pentacle
- new `id`
- new `screen`
- new tags
- nested CSS/image assets
- changed HTML layout
- at least one unknown manifest field

Publish it through the machine publisher. An already-running Pentacle instance must discover and preview it from hub data without a Pentacle git diff, rebuild, redeploy, or restart. Unknown fields must not affect browsing.

## Cross-Machine Validation Matrix

Final acceptance requires at least one successful publish/view proof for each machine as a publisher and each machine as a viewer:

| Publisher | Viewer: Bartimaeus | Viewer: Merlin | Viewer: Amaterasu |
| --- | --- | --- | --- |
| Bartimaeus | same-machine acceptable | required | required |
| Merlin | required | same-machine acceptable | required |
| Amaterasu | required | required | same-machine acceptable |

If a machine is temporarily unavailable, the missing matrix entries must be reported as residual risk and the feature cannot be marked fully accepted.

## Validation Plan

- Unit-test manifest validation and path-safety logic in the publisher/hub code.
- Unit-test Pentacle dashboard data normalization for hub index, stale hub data, empty index, and local fallback.
- Generate the Pentacle mobile chat QA artifact and verify it appears in the local fallback.
- Publish one artifact to Dashboard Hub from Merlin and verify its `entryUrl` loads from a second machine.
- Publish one artifact from Amaterasu and verify it appears on Merlin.
- Verify loading, empty, stale, error, latest-first sorting, filtering, phone-sized preview, desktop-sized preview, and Open button behavior with screenshots or recorded manual evidence.
- Redeploy/restart Pentacle where source changes are required and verify a fresh startup log.

## Rollout

1. Land the generic Pentacle dashboard with local fallback and hub-index consumption.
2. Add Dashboard Hub upload/static-serving/index support.
3. Add a machine-local publisher script/service.
4. Convert `pentacle-mobile` QA output to `.ui-review/<artifact>/manifest.json`.
5. Deploy publisher and Pentacle updates to Bartimaeus, Merlin, and Amaterasu.
6. After deployed, new artifact content changes require only repo generator/publisher runs, not Pentacle app updates.

## Rollback

- Disable publisher service on a machine to stop new uploads.
- Keep Pentacle local fallback available for development.
- If hub index is malformed, Pentacle should show an error and not break other dashboards.
